# SwiftReporter MCP workflows

## General decision sequence

1. **Establish access**: call `account_get_entitlements` when the task may depend on role, organisation, Back of House, Payments, plan, or account permissions.
2. **Discover**: list/search records; paginate when a cursor is returned.
3. **Disambiguate**: if multiple records match, present names/IDs and ask the user to choose.
4. **Read**: fetch the exact target and related parent records.
5. **Validate**: repeat important target, recipient, dates, amounts, or impact.
6. **Act**: call the narrowest mutation tool.
7. **Verify**: read the result or poll the job; clearly report any partial failure.

Never silently continue from one user's or organisation's result into another context.

## Inspect an inspection

1. `inspections_list` using active/archived filter and pagination.
2. Resolve the intended inspection.
3. `inspections_get`; use `full: false` first.
4. Use `observations_list` and `media_list` only if the user needs those details.
5. Use `observations_get` for specific finding text/evidence.

Avoid fetching full nested inspection data when a summary answers the question.

## Create or update an inspection

1. Confirm the user's active template/intent; `inspections_create` uses the active template.
2. Validate name, client, and address.
3. Call `inspections_create`; retain the returned inspection ID.
4. Before updates, `inspections_get` the exact ID.
5. Call `inspections_update` with only explicitly requested fields.
6. Read back the inspection if verification matters.

Do not overwrite fields the user did not request.

## Work with observations

### Create

1. Read the parent inspection.
2. Confirm category/subcategory and finding wording if supplied.
3. `observations_create`.
4. Use `observations_get` to verify.

### Summarize

1. Read the observation and parent inspection.
2. If Custom Prompts or Glossary affect the requested output, read the relevant settings.
3. Call `observations_summarize` with matching observation and inspection IDs.
4. Explain that the output was generated by the same web summarization workflow and should be reviewed.

### Delete

1. `observations_get`.
2. State the observation title, parent inspection, and that deletion is permanent.
3. Obtain explicit confirmation.
4. Call `observations_delete`.

## Generate a report

1. Read the inspection and ensure the requested report target is correct.
2. Call `reports_generate_start`.
3. Preserve and report the `job_id`.
4. Poll `reports_job_status` with reasonable intervals until `succeeded` or `failed`.
5. Return the result on success; on failure, return the tool error without inventing a download URL.

Do not repeatedly start duplicate report jobs while one is queued/running.

## Share an inspection

1. Read the inspection and verify the current user is its owner.
2. Validate and repeat the recipient email.
3. Explain that an email invite will be sent.
4. Obtain explicit approval.
5. Call `inspections_share_send` once.

## Template and comment workflows

- Use `templates_list` → `templates_get` before copying or describing structure.
- `templates_create` creates an empty template; it does not create sections or run the AI wizard.
- Use `templates_copy_prebuilt` only after identifying the exact prebuilt template ID.
- Search comments before creating a near-duplicate.
- A comment is reusable library content; to add a finding to an inspection, use `observations_create`.

## Contacts and appointments

### Merge contacts

1. Confirm BOH entitlement.
2. List contacts and identify the exact primary and duplicate IDs.
3. Summarize which record survives and that references may be consolidated.
4. Obtain explicit confirmation.
5. Call `contacts_merge` once.

### Create appointment

1. Confirm title and timezone-aware start/end values with the user.
2. Ensure end is after start.
3. Call `appointments_create`.
4. Report the appointment ID and normalized times.

## Contract and invoice external actions

### Send contract

1. `contracts_list` and resolve the exact contract.
2. Resolve/verify the intended contact ID when supplied.
3. State that a signing request will be sent externally.
4. Obtain explicit approval.
5. Call `contracts_send` once.

### Create/send invoice

1. Confirm Payments access through `account_get_entitlements`.
2. Use `invoices_list` to avoid duplicate billing.
3. For creation, use only fields known from the user's stated invoice or current web workflow; never invent amount, currency, recipient, line items, due date, or tax.
4. Before `invoices_send`, repeat invoice and recipients and obtain approval.
5. Call each external action once and report the returned status.

### Void invoice

1. Resolve and read the invoice.
2. Call `invoices_prepare_void`.
3. Show the exact `impactSummary` and ask the user to confirm.
4. Only then call `invoices_confirm_void` with the returned confirmation value.
5. If the confirmation is no longer valid, prepare the void again.

## Settings

- Profile and notification changes: summarize changed fields before calling the update tool.
- Custom Prompt changes affect future AI summaries; show the new text and obtain approval.
- Glossary upsert: search/list first to avoid duplicates; update only an owned `term_id`.
- Quote Calculator and booking settings are read-only through current MCP tools.

## When to send the user to the web UI

Use the SwiftReporter web UI for organisation member/admin changes, account/organisation deletion, Payments integrations and billing portals, AI template creation, public booking completion, and password/email/security changes.
